## Bug 16 — Phi-4 rambling / fake thinking on instruct models

**Evidence:** Log lines 1982–2594. `templateSupportsThinking=false` but `enable_thinking=true`, `Thinking budget: 2048`. Run-on garbage, 0 tool calls, context creep to 76%.

**Cause:** Blind `enable_thinking` pass when profile `thinkTokens.mode !== 'none'` without template check. Phi profile had `mode: 'budget'`. User `reasoningEffort` applied thought budget globally.

**Root fix:** [x] Gate `enable_thinking` on `_templateSupportsThinking`. [x] Phi profile → `mode: 'none'`. [x] `_thinkingCapable` disables thought budget at generation. [ ] Phi SWA disabled in bundled llama.cpp (upstream, separate).

## Bug 1 — Gemma 4: `unknown model architecture: 'gemma4'`

**Evidence:** Log line 3837–3839. Installed binary at `C:\Program Files\guIDE\...` rejects gemma4.

**Cause:** The shipped llama.cpp native addon was compiled from a prebuilt bundle that predates gemma4 architecture support. Metadata file says `b9253` but the **binary** does not match — same class of failure as v0.3.70 (metadata updated without recompiling). CI attempts from v0.3.70–v0.84 to download/build newer llama.cpp **failed** on GitHub Actions (compilation, path layout, git init, rate limits). v0.3.85 **reverted** to prebuilt binaries and explicitly deferred Gemma 4.

**Not impossible.** Blocked on shipping a llama.cpp binary that includes gemma4. Ollama/LM Studio ship their own updated llama.cpp with each release.

**Root fix (not a guIDE pipeline hack):**
1. Wait for `node-llama-cpp` npm release whose **prebuilt** binaries include gemma4 (track [node-llama-cpp releases](https://github.com/withcatai/node-llama-cpp/releases)), OR
2. Fix CI to reliably run `npx node-llama-cpp source build` on all 5 platform jobs with correct source layout — the v0.3.84 chain got closest but still failed on Windows CPU build step.

**What NOT to do:** Regex/arch detection for gemma4, model-specific load paths, or metadata-only tarball swaps without compiling the addon.

## Bug 2 — GLM double responses (thinking duplicated in chat)

**Status:** [x] segmentType routing implemented — [ ] raw think parser removal pending verify

**Evidence:** Log lines 172, 218, 232–249. User message at line 210 explicitly reports this. Raw model output contains the same prose before AND after `</think>`.

**Cause — full pipeline:**

1. GLM loads with `enable_thinking=true`, `chatWrapper: JinjaTemplate` (line 182).
2. **B4 native segment API does NOT activate** for GLM (no `✓ B4: Native onResponseChunk segment API active` log line).
3. Instead, `_sfProcessChunk` raw-text path fires `orphan </think> consumed — retroactively marking preceding text as thinking` (lines 1142–1238 in chatEngine.js).
4. That retroactive handler is **post-hoc remediation** — it moves already-forwarded text into the thinking panel after the fact.
5. Model then generates the same answer again after the close tag as visible response.
6. ChatPanel.jsx (lines 1404–1408) already excludes thinking from `messageContent` — but if retroactive move fails or both chunks land as `text` segments, user still sees duplication.

**How Ollama/LM Studio handle it:** GLM Jinja template separates thinking from response at template level. `enable_thinking` controls whether thinking blocks are emitted. Content before/after `</think>` is parsed by the template, not by a streaming regex filter.

**Root fix:**
1. Fix `onResponseChunk` handler (chatEngine.js ~1704) to route by `chunk.segmentType` — thinking segments → `llm-thinking-token`, content segments → normal `onToken` path. Currently **all** segment chunks are treated as thinking (B4 bug).
2. Once segment routing works, GLM with `enable_thinking=true` should get native thinking separation from node-llama-cpp's Jinja wrapper — verify with GLM-4.6V after step 1.
3. **Remove** the entire `_sfProcessChunk` raw `<think>` / orphan-close-tag detection path once native segments work for all thinking models. That ~200-line regex state machine is the band-aid.

**Research:** [node-llama-cpp LlamaChatResponseSegmentChunk](https://node-llama-cpp.withcat.ai/api/type-aliases/LlamaChatResponseSegmentChunk) defines `segmentType: ChatModelSegmentType`. DeepSeek R1 support commit #428 added proper segment separation.

## Bug 4 — Context collapse on 4GB VRAM

**Status:** [x] `computeUnifiedVramBudget` — single pass for gpuLayers + contextSize (pre-load + post-load refine)

**Evidence:** Multiple models:
- DeepSeek 1.5B: ctx 55552 → 42% of desired (line 100)
- GLM 4.6V: ctx 15360 → 11.7% (line 142)
- gpt-oss 20B: ctx 14592 → 11.1% (line 286), Fix D cap logged (line 284)
- Qwen3.6 35B: ctx **5120** → 7.8% (line 711) — severe collapse
- Llama 3.2 3B: ctx 7936 → 6.4% (line 1337)

**Cause:** GPU layer allocation and context sizing are **two separate passes** that don't share a single VRAM budget:
1. GPU layers computed with MIN_VIABLE_CONTEXT=4096 KV reservation (lines 477–491) — good improvement but still prioritizes layers.
2. Post-load context computed from remaining VRAM (lines 542–549).
3. Fix D further caps context when gpuRatio < 0.3 (lines 558–563) — drops Qwen 35B to ~5K.
4. On 4GB GPU with 20B+ models, almost all VRAM goes to partial GPU layers; KV cache gets scraps.

**How Ollama handles it:** PR #1850 — layer offloading accounts for KV cache size in the same calculation. Default context scales with VRAM (<24GB → 4K default).

**Root fix:** Single unified VRAM budget function (one pass, not two):
```
available_vram = vramFree
for each candidate (gpuLayers, contextSize):
  model_vram = gpuLayers * bytesPerLayer
  kv_vram = kvBytesPerToken * contextSize * (gpuLayers / totalLayers)
  if model_vram + kv_vram + overhead <= available_vram: accept
pick the (gpuLayers, contextSize) pair that maximizes context while keeping gpuLayers > 0
```
No hardcoded MIN_VIABLE_CONTEXT=4096 — derive minimum from `kvBytesPerToken * trainMinContext` where trainMinContext comes from model metadata. Formula must work on 4GB and 128GB equally.

**Symptom chain:** Bug 4 causes Bug 7 (tool prompt eats 28–37% of tiny context), Bug 10 (models chat instead of tool-calling), and generation-space trimming (lines 1771–1814).

## Bug 5 — Tool prompt consumes disproportionate context

**Status:** [x] `buildBudgetProportionalToolPrompt` — budget from ctx − history − user − gen reserve; category parts from `getCompactToolHint()`

**Evidence:**
- Full prompt: 15077 chars when ctx ≥ 8192 (lines 166, 212, 304, 3472)
- Compact prompt: 2625 chars when ctx < 8192 (lines 1285, 1355, 1394)

**Cause:** `chatEngine.js` line 975: `useCompact = compactToolPrompt && (contextTokens < 8192 || toolPct > 40)`. When context is 15360+, full 69-tool catalog is injected even though 4300 tokens of tools in a 15K window leaves little room for conversation. Tools are **never** passed to `generateResponse({ functions })` — only text-appended to system prompt (line 1830 has no functions in genOptions).

**Root fix:** Size tool catalog to **remaining prompt budget** at generation time:
- Compute `promptBudget = contextSize - MIN_GENERATION_RESERVE - historyTokens - systemPromptTokens`
- Append tool categories from `getCompactToolHint()` until budget exhausted (mcpToolServer already returns array of parts for this — line 3358)
- Remove hardcoded 8192 threshold — budget is computed from actual context and history

Long-term: evaluate node-llama-cpp native `functions` parameter for models whose templates support tool calling, with text-prompt fallback only when template lacks tool support. This eliminates the 15K char injection entirely for supported models.

## Bug 6 — Missing `inject-user-message` IPC handler

**Status:** [x] Fixed in `electron-main.js`

**Evidence:** Log lines 1275, 2426, 3366: `Error: No handler registered for 'inject-user-message'`.

**Cause:** `preload.js` line 64 exposes `injectUserMessage` via `ipcRenderer.invoke('inject-user-message')`. `ChatPanel.jsx` line 1026 calls this when user sends a message during streaming. `chatEngine.js` line 2619 implements `injectUserMessage()`. **electron-main.js has no `ipcMain.handle('inject-user-message')`** — only the HTTP API path would work, and ChatPanel uses IPC.

**Root fix:** Add to electron-main.js:
```js
ipcMain.handle('inject-user-message', (_e, { text }) => {
  llmEngine.injectUserMessage(text);
  return { success: true };
});
```
Wire the same in the HTTP handler if browser mode uses fetch. One line, end-to-end.

## Bug 7 — Model load race / "Model not ready"

**Status:** [x] `_loadState` machine, `waitForReady()`, chat gate, session/clear awaits load — [ ] beforeExit listener leak (not in guIDE code)

**Evidence:**
- Line 129: `Failed to load model` during concurrent session/clear + model switch (lines 113–128)
- Line 3450: `ai-chat local ERROR: Model not ready` — chat sent 1.6s after model switch started (lines 3433–3450)
- Line 3443: `MaxListenersExceededWarning: 11 beforeExit listeners` — repeated dispose/load cycles leaking listeners

**Cause:**
1. User switches model → `_dispose` starts → user triggers session/clear or sends chat before `isReady=true`.
2. `initialize()` queue (lines 283–294) waits for prior load but session/clear calls `cancelGeneration` + `resetSession` without coordinating with in-flight `_doInitialize`.
3. Frontend does not gate send on `modelLoading`/`isReady` in all paths.

**Root fix:**
1. Add load-state machine: `idle | loading | ready | disposing`. Reject or queue chat requests during `loading`/`disposing` with explicit user feedback (not silent error).
2. `session/clear` must await in-flight model load completion before `resetSession`, or cancel load cleanly.
3. Fix `beforeExit` listener leak — register once in app startup, not per ChatEngine dispose.

## Bug 9 — Vision mmproj false matches

**Evidence:** Lines 105–108, 147–148, 291–293: `StaffPix/mmproj-F32.gguf` matched to DeepSeek, Llama, gpt-oss — unrelated models.

**Cause:** `_readGgufEmbdLength` returns `unknown` for these models (line 105). Without embedding length, mmproj validation is skipped (visionServer.js line 565: `if (modelEmbdLen)` guard).

**Root fix:** Read `embedding_length` from GGUF architecture metadata already parsed at model load (`chatEngine.js` line 351 logs it). Pass `modelInfo.embeddingLength` to VisionServer at load time. Reject mmproj when dimensions don't match. When unknown, do **not** match random sibling mmproj — report vision unavailable.

## Bug 11 — Continuation after tools may duplicate content

**Evidence:** Lines 644–658. gpt-oss continuation after 10 successful tool calls generates 613 chars that repeat the folder structure narrative already in turn 1.

**Cause:** Tool results + prior assistant message already describe completed work. Continuation prompt does not tell model that tools **already executed** — model re-narrates.

**Root fix:** Continuation message builder must state execution status: "Tools listed below have ALREADY been executed. Do not repeat actions. Summarize results or proceed to next step." This is prompt content at the architectural level (continuation builder), not post-hoc deduplication.
